# djinni-parser

Особистий скрипт для збору публічних вакансій Frontend (React/JavaScript/TypeScript)
з [Djinni.co](https://djinni.co) у CSV — для власного пошуку роботи. Не для
комерційного використання чи перепродажу даних.

## Дотримання правил сайту

- Програмно перевіряє `djinni.co/robots.txt` перед кожним запитом
  (`urllib.robotparser`), а не покладається на захардкожений список шляхів.
  Якщо сайт зробить редирект — перевіряється і кінцевий URL, а не лише той,
  який запитали (`requests` сам іде за редиректами, тож без цього одна
  перевірка легко пропустила б заборонений шлях).
- Ходить лише на `/jobs/<id>-<slug>/` та `/jobs/?...` (пошук/пагінація).
- Випадкова затримка 2-4с між запитами, без паралельних запитів, звичайний
  User-Agent браузера, без логіну/авторизації.
- Зупиняється одразу, якщо сайт сигналізує блокування (HTTP 429/403/503,
  CAPTCHA-сторінка) — не намагається "пробитись" повторними спробами. Це
  відрізняється від тимчасових збоїв (timeout, 5xx): їх скрипт **повторює**
  до 3 разів з експоненційною паузою (1с, 2с, 4с), бо це, найімовірніше,
  випадковий мережевий глюк, а не сигнал сайту "припиніть сканування".

**robots.txt — це технічна чемність до сканерів, а не заміна Terms of Service.**
Перед активним використанням варто самостійно ознайомитись з ToS Djinni.

## Встановлення

```
python -m venv .venv
.venv\Scripts\activate
pip install -r requirements.txt
```

## Використання

```
python djinni_parser.py --dry-run              # лише порахувати вакансії, нічого не зберігати
python djinni_parser.py                        # повний збір у vacancies_frontend.csv
python djinni_parser.py --max-pages 5           # обмежити глибину пагінації на ключове слово
python djinni_parser.py --output my.csv         # інший файл виводу
python djinni_parser.py --delay-min 4 --delay-max 7   # довша затримка між запитами (за замовчуванням 2-4с)
python djinni_parser.py --start-page 15 --max-pages 20   # продовжити пагінацію з конкретної сторінки
```

Якщо прогін впав через блокування сайту (CAPTCHA/429/403/503) під час пошуку —
`--start-page` дозволяє продовжити з тієї сторінки, де зупинились, замість
повторного проходу з першої сторінки на кожне ключове слово (що саме й
витрачає бюджет запитів і наближає наступне блокування). Застереження:
сортування 1-ї сторінки Djinni може зміщуватись між прогонами (нові вакансії
підіймаються нагору), тож високий `--start-page` ризикує пропустити кілька
щойно опублікованих вакансій, які тепер потрапили б на вже пройдені сторінки.

Повторний запуск дозаписує лише нові вакансії — вже наявні в CSV (за `url`)
пропускаються, тож перерваний прогін безпечно продовжити повторним запуском
тієї самої команди.

Якщо в конкретної вакансії несподівана структура даних (Djinni іноді відхиляється
від очікуваної схеми — так уже було двічі), скрипт друкує `[error] failed to
parse ...` і йде далі, а не падає повністю: втрачається одна вакансія, а не
весь накопичений прогрес прогону.

## Що саме збирається

Дані витягуються переважно зі структурованого блоку `<script type="application/ld+json">`
(schema.org `JobPosting`), який Djinni вбудовує в кожну сторінку вакансії — це
надійніше за парсинг HTML-верстки, яка може змінюватись. З цього ж блоку
береться `title`, `company`, `description`, `posted_date`, `salary_from/to`
(коли роботодавець його вказав) та `experienceRequirements`.

Поля, яких немає у структурованих даних, розпізнаються евристично з тексту
опису (регулярки, ключові слова) — і мають відповідні обмеження:

- **`salary_from`/`salary_to`** — здебільшого будуть порожні: більшість
  роботодавців на Djinni не вказують точну вилку.
- **`english_level`** — береться напряму зі структурованого блоку "Вимоги до
  володіння мовами" на сторінці вакансії (`span.csc--language`, поза
  ld+json та поза текстом опису), у форматі CEFR ("B2 – Вище середнього").
  Це набагато надійніше за вгадування з вільного тексту: на реальному зборі
  331 вакансії текстова евристика губила ~60% ("не вказано"), тоді як
  структуроване поле дає лише ~9%. Вільнотекстовий розпізнавач
  (`infer_english_level`) лишається лише як fallback для рідкісної сторінки
  без цього блоку.
- **`experience_level`** — спершу шукає явне слово (junior/middle/senior/
  trainee) **лише в заголовку** вакансії (не в описі — опис часто згадує
  чужий рівень, напр. "working alongside Senior developers" про менторів
  команди, що раніше хибно перезаписувало реальний рівень вакансії). Якщо
  заголовок мовчить — оцінює за роками досвіду зі структурованого блоку
  сторінки ("Виключно від 2 років досвіду" / "Без досвіду"), а якщо і його
  немає — за `monthsOfExperience` з ld+json.
- **`work_format`** — береться зі структурованого поля сторінки, яке є
  мультивибором: вакансія може одночасно приймати кілька форматів, тоді
  значення виглядає як `"hybrid, office"` (а не одне з них навмання). Якщо
  структурованого поля немає — fallback на `jobLocationType == "TELECOMMUTE"`
  з ld+json, потім на текст опису.

Пізніше додані поля (той самий принцип "структуроване спершу, евристика як
fallback"):

- **`category`** — Frontend/Fullstack/Other, найкраще припущення за
  заголовком і описом. Ролі поза фронтендом (QA/PM/SEO/чистий бекенд), що
  збіглись за ключовим словом пошуку, і React Native (мобільна розробка)
  свідомо потрапляють в "Other" — жодна вакансія не викидається з виводу,
  "Other" лише позначає її як шум для подальшої фільтрації.
- **`tech_stack`** — фіксований словник технологій (React, TypeScript,
  Docker, CI/CD тощо), знайдений у тексті опису; вакансія може мати кілька.
- **`employment_type`**/**`industry`** — зі структурованого блоку ld+json,
  коли Djinni його заповнює (на практиці `industry` майже завжди порожнє).
- **`valid_through`** — дата закінчення дії вакансії з ld+json, якщо є.
- **`salary_usd_normalized`** — грубе перетворення `salary_from`/`salary_to`
  в USD за фіксованим курсом (не live-курс) лише для сортування/фільтрації
  на одній шкалі.
- **`must_have`**/**`nice_to_have`** — евристичний розділ тексту опису за
  заголовками секцій ("Вимоги"/"Буде перевагою"/"Requirements"/"Nice to
  have" тощо, українською й англійською) — найкраще припущення, не всі
  вакансії структуровані настільки чітко, щоб розділ спрацював.

## Обмеження

CSS-класи/структура HTML сторінки можуть з часом змінитись — зокрема
`span.csc--language` (`english_level`) та картка `div.card.card-body` зі
словом "досвід" (`experience_level`/`work_format`) — якщо парсинг
почне повертати багато порожніх полів, варто звірити актуальну розмітку
сторінки вакансії з кодом у `djinni_parser.py`.
